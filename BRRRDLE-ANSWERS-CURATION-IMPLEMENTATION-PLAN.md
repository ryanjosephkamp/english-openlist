## Scope

Implement the new Brrrdle `answers` curation algorithm while preserving the complete per-length `validGuesses` source list.

This plan intentionally limits implementation changes to the Brrrdle artifact generation path:

- `scripts/generate_brrrdle_artifacts.py`
- `tests/test_generate_brrrdle_artifacts.py`
- `.github/workflows/daily_update.yml`, only if its verification step must validate the new JSON shape
- Brrrdle-specific documentation generated by or describing that pipeline, only if needed
- A new change-log entry in `change-logs/`

Do not delete, rename, or remove existing files or existing source-list logic. Do not alter unrelated dictionary validation, data update, Hugging Face upload, or blog-generation behavior.

## Current state reviewed

- `scripts/generate_brrrdle_artifacts.py` currently normalizes lowercase ASCII alphabetic words, groups them by length 2 through 35, and writes one `words_length_N.json` file per length.
- Current per-length files are JSON arrays of word entries, with `word` always present and `definition` included only when available.
- The daily workflow already runs `scripts/generate_brrrdle_artifacts.py` and verifies that all 34 `words_length_N.json` files exist before upload.
- Existing Brrrdle documentation describes `words_length_N.json` as the primary artifacts and notes the transitional length-5 legacy files.
- `BRRRDLE-ANSWERS-CURATION-SPEC.md` now requires each per-length file to include `metadata.curation`, `answers`, and `validGuesses`.

## Implementation principles

1. Preserve `validGuesses` as the complete sorted list for each length.
2. Add `answers` as a curated subset selected from `validGuesses`.
3. Keep all filtering rules that determine the valid per-length source list unchanged.
4. Keep generation deterministic by using `seed = 42 + length`.
5. Keep output generation inside the existing preprocessing script unless a small helper module is clearly cleaner.
6. Keep the daily workflow command unchanged if possible; update only its verification checks if needed.
7. Keep legacy length-5 compatibility outputs unless a later approved task removes them.

## Phase 1 — Add curation helpers to the existing generator

Add helper functions in `scripts/generate_brrrdle_artifacts.py` near the existing Brrrdle generation helpers.

Required helpers:

- `get_target_sample_size(total_valid_guesses: int) -> int`
  - If `total_valid_guesses < 1000`, return `total_valid_guesses`.
  - Otherwise return `max(2000, min(8000, int(math.sqrt(total_valid_guesses) * 22)))`.
- `build_curation_metadata(length: int, target_sample_size: int, curation_date: datetime) -> dict[str, Any]`
  - `method`: `stratified_quality_score_v1`
  - `seed`: `42 + length`
  - `target_sample_size`: computed target
  - `curation_date`: UTC ISO timestamp ending in `Z`
  - `note`: `Stratified by starting letter + quality score (frequency 0.45, positional 0.30, vowel balance 0.15, uniqueness 0.10)`
- `quality_score(word: str, corpus_for_length: list[str], positional_stats: ..., letter_stats: ...) -> float`
  - Combine normalized component scores using the exact approved weights:
    - `0.45` letter frequency similarity
    - `0.30` positional match score
    - `0.15` vowel/consonant balance score
    - `0.10` uniqueness score
- `select_curated_answers(valid_words: list[str], length: int) -> tuple[list[str], int]`
  - Return all words unchanged for lists under 1000.
  - For lists of 1000 or more, stratify by starting letter and select the highest-scoring words from each bucket.

Because the visible spec references a previously supplied full helper implementation that is not present in the repository, define each component score in the implementation comments/tests before coding:

- Letter frequency similarity: compare the word's letter distribution with the full per-length corpus letter distribution and normalize to `[0, 1]`.
- Positional match score: compare each character against the per-position frequency distribution for that length and average normalized positional frequencies.
- Vowel/consonant balance score: reward words whose vowel ratio is closest to the full per-length corpus vowel ratio.
- Uniqueness score: reward words with more unique letters, normalized by word length.

Tie-breaking must be deterministic. Use a local `random.Random(42 + length)` instance only for stable tie ordering inside equal-score groups; do not mutate global random state.

## Phase 2 — Create the new per-length JSON payload shape

Add a focused payload builder in `scripts/generate_brrrdle_artifacts.py`.

Planned function:

- `create_word_list_json(valid_guesses: list[str], length: int, generated_at: datetime) -> dict[str, Any]`

Behavior:

1. Use the sorted words from the current full per-length list.
2. Select curated answer words using `select_curated_answers`.
3. Preserve `validGuesses` as the complete current full word-string list for that length.
4. Populate `answers` as the selected subset of word strings.
5. Add `metadata.curation` exactly as required by the spec.

The output object for each `words_length_N.json` should be:

```json
{
  "metadata": {
    "curation": {
      "method": "stratified_quality_score_v1",
      "seed": 42,
      "target_sample_size": 2000,
      "curation_date": "2026-05-27T00:00:00Z",
      "note": "Stratified by starting letter + quality score (frequency 0.45, positional 0.30, vowel balance 0.15, uniqueness 0.10)"
    }
  },
  "answers": [],
  "validGuesses": []
}
```

Use the actual seed `42 + N` for each length. Use the existing generation timestamp as `curation_date` so all artifacts from one run share a reproducible run timestamp.
